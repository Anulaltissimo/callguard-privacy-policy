Informativa Privacy — CallGuard (FASE 1, solo on-device)

Questa pagina descrive ciò che è attivo oggi nell’app. In FASE 1 non esistono backend/telemetria.

1) Titolare del trattamento / Contatti

Sviluppatore: AnuGodos
Email supporto/privacy: clanity184@gmail.com

2) Dati trattati nell’app (FASE 1)

CallGuard tratta dati localmente sul dispositivo per fornire le funzionalità di filtro chiamate e gestione regole.

Categorie tipiche (salvate localmente):

  • Numeri di telefono e identificativi chiamante (per applicare le regole, ad esempio “non in contatti”).
  • Eventi chiamata (es. data/ora, esito: consentita/bloccata, eventuale invio SMS di cortesia se attivo).
  • Regole e preferenze (schedule, impostazioni guardia, preferenze SIM, template SMS, impostazioni app).
  • Contatti (solo per verifica): l’app può leggere la rubrica per determinare se un numero è presente nei contatti.

2.b) Backup locale cifrato (opzionale)

  • Se abiliti il backup e imposti una password, CallGuard può creare un file di backup cifrato salvato localmente nella cartella Download del dispositivo (es. CallGuardBackup_v1.cgbak).
  • Il backup può includere impostazioni/regole/scheduli/template e una porzione di eventi chiamata recenti (ad oggi fino a ~30 giorni e con un limite massimo).

3) Dove vanno i dati

  • FASE 1: nessun backend / nessun invio dati verso un server da parte di CallGuard.
  • FASE 1: nessuna telemetria (non è implementata/abilitata).

4) Permessi Android e perché

CallGuard richiede alcuni permessi/ruoli per funzionare:

  • Call screening (ruolo di filtro chiamate): consente all’app di valutare e decidere localmente se consentire o bloccare una chiamata in base alle regole definite dall’utente. CallGuard non registra audio e non accede al contenuto delle conversazioni.
  • READ_CONTACTS (lettura contatti): necessario per applicare regole come “blocca chiamate non in contatti”, confrontando localmente il numero chiamante con la rubrica.
    • Se neghi questo permesso, CallGuard può continuare a funzionare, ma le regole basate sui contatti potrebbero non essere accurate.
  • SEND_SMS (invio SMS) (opzionale): usato solo se abiliti la funzione di SMS di cortesia (auto-reply) quando una chiamata viene bloccata. Puoi disattivare questa funzione in qualsiasi momento.
    • CallGuard non legge né intercetta gli SMS in ingresso e non richiede permessi di lettura/ricezione SMS per funzionare.
  • POST_NOTIFICATIONS (opzionale): per notifiche locali sullo stato o sugli esiti (se presenti e abilitate).

5) Base giuridica

Esecuzione del servizio richiesto dall’utente: filtro chiamate, regole, registrazione eventi locali e (se attivo) SMS di cortesia.

6) Conservazione e cancellazione (FASE 1)

  • I dati sono conservati localmente sul dispositivo.
  • Puoi cancellare i dati rimuovendo i contenuti dall’app (se la UI lo consente) oppure usando le opzioni di sistema Android (es. Cancella dati / disinstallazione).
  • Se hai attivato il backup, puoi cancellare anche i file di backup dalla cartella Download (tramite file manager Android).

7) Condivisione con terze parti

FASE 1: CallGuard non condivide dati con terze parti.

8) Sicurezza

CallGuard minimizza i dati trattati e opera al momento esclusivamente on-device.

9) Modifiche a questa informativa

Potremmo aggiornare questa informativa in futuro (ad esempio in caso di introduzione di telemetria opt-in o servizi cloud). In tal caso, aggiorneremo la data e pubblicheremo la versione aggiornata.

Privacy Policy — CallGuard (PHASE 1, on-device only)

This page describes what is enabled today in the app. In PHASE 1 there is no backend/telemetry.

1) Data Controller / Contact

Developer: AnuGodos
Support/Privacy email: clanity184@gmail.com

2) Data processed in the app (PHASE 1)

CallGuard processes data locally on the device to provide call screening and rule management.

Typical categories stored locally:

  • Phone numbers / caller identifiers (to apply rules, e.g., “not in contacts”).
  • Call events (e.g., date/time, outcome: allowed/blocked, optional courtesy SMS sent if enabled).
  • Rules and preferences (schedules, guard settings, SIM preferences, SMS templates, app settings).
  • Contacts (lookup only): the app may read the address book to determine whether a number is in contacts.

2.b) Local encrypted backups (optional)

  • If you enable backups and set a password, CallGuard may create an encrypted backup file stored locally in the device Downloads folder (e.g., CallGuardBackup_v1.cgbak).
  • The backup may include settings/rules/schedules/templates and a portion of recent call events (currently up to ~30 days, with a maximum limit).

3) Where data goes

  • PHASE 1: no backend / no server upload performed by CallGuard.
  • PHASE 1: no telemetry (not implemented/enabled).

4) Android permissions and why

CallGuard requires certain permissions/roles to work:

  • Call screening role: allows the app to decide locally whether to allow or block an incoming call based on user-defined rules. CallGuard does not record call audio and does not access voice content.
  • READ_CONTACTS: required to apply rules such as “block calls not in contacts” by matching the incoming number against the local address book.
    • If you deny this permission, CallGuard can still run, but contact-based rules may not work accurately.
  • SEND_SMS (optional): used only if you enable courtesy SMS auto-replies when a call is blocked. You can disable this feature at any time.
    • CallGuard does not read or intercept incoming SMS messages and does not require SMS read/receive permissions to function.
  • POST_NOTIFICATIONS (optional): for local notifications about status/outcomes (if present and enabled).

5) Legal basis

Providing the service requested by the user: call screening, rules, local event logging, and (if enabled) courtesy SMS.

6) Retention and deletion (PHASE 1)

  • Data is stored locally on the device.
  • You can delete data from within the app (when available) or via Android system options (e.g., Clear storage / uninstall).
  • If you enabled backups, you can also delete backup files from the Downloads folder (using the Android file manager).

7) Sharing with third parties

PHASE 1: CallGuard does not share data with third parties.

8) Security

CallGuard minimises data processing and currently operates exclusively on-device.

9) Changes to this policy

We may update this policy in the future (for example, if opt-in telemetry or cloud services are introduced). In that case, we will update the date and publish the updated information.
